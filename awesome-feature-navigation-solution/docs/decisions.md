# Decision Log

## 2026-04-18 — Исправление самопересечений траектории

### Проблема
Сырые круги (`trajectory_raw.csv`) содержали самопересечения, которых в реальности не было. Алгоритм усреднения по кругам частично их сглаживал, но Lap 3 отбрасывался как аномальный.

### Анализ причин

1. **Дрейф yaw от IMU** — гироскоп интегрируется покадрово. Начальный bias компенсируется через `imu_yaw_bias_window_sec` в `imu_io.py`, но температурный дрейф во время движения не устраняется. При прямолинейном следовании по линии `angle_rad ≈ 0`, поэтому angle-коррекция (`vision_yaw_gain`) не срабатывает, а боковое смещение (`bottom_x`) растёт.

2. **Боковая коррекция не исправляет курс** — P-регулятор на `bottom_x` сдвигает оценку позиции, но не yaw. Медленный дрейф курса этим не компенсируется.

3. **Постоянная скорость** — `forward_speed_mps: 0.25` берётся константой, а не извлекается из сенсоров. Это **известное ограничение алгоритма**, не баг: IMU+камера в принципе достаточны для оценки скорости (акселерометр, оптический поток, известный масштаб объектов), но это не реализовано в текущем пайплайне.

### Принципиальная граница алгоритма

Алгоритм должен строить траекторию **только из IMU + кадры**. Всё, что мы добавляем в код, должно быть одним из:
- **Интегрирование сенсора** (yaw из гироскопа, положение из акселерометра).
- **Коррекция оценки по наблюдению** (камера видит ленту → исправляем накопленную ошибку IMU).

Нельзя вкладывать в код **кинематические предположения о роботе** (например, "робот замедляется на поворотах") — это лишняя априорная информация, которая может не соответствовать реальности.

### Принятые решения

#### ✅ Yaw-коррекция из бокового смещения (`vision_lat_yaw_gain`)
**Файл:** `awesome_feature_navigation/trajectory.py`

```python
yaw -= k_lat_yaw * err_px * dt_frame
```

Это **легитимная сенсорная фьюжн**: устойчивое боковое смещение линии (`bottom_x` от центра) — это наблюдаемое следствие ошибки курса, а не ошибки позиции. Если робот следует по прямой ленте, но `bottom_x` стабильно смещён — значит накопился bias по yaw. Камера исправляет известный источник ошибки IMU.

По физике: при смещении 100px и `k_lat_yaw = 0.0001` коррекция ~0.01 рад/с, что соразмерно типичному дрейфу MEMS-гироскопа (0.001–0.01 рад/с).

Параметр: `vision_lat_yaw_gain` (default: `0.0001`).

#### ❌ ОТКАЧЕНО: модуляция скорости от yaw_rate
**Причина отката:** `speed_factor = 1 - k * |yaw_rate|` — это выдуманная кинематика "робот замедляется на поворотах". Это **предположение о роботе**, а не вывод из данных. Для дифф-привода с контролем скорости оно неверно. Такие эвристики не должны быть в алгоритме реконструкции траектории.

Если `forward_speed_mps` неверен — это надо решать **извлечением скорости из сенсоров**, а не подгонкой.

### Нереализованные улучшения (правильные пути для speed)

- **IMU translation** (`imu_use_translation: False`) — интеграл акселерометра после удаления гравитации. Работает плохо без ZUPT (zero-velocity updates).
- **Оптический поток** между кадрами — прямой сенсорный источник скорости.
- **Масштаб ленты** в кадре при известной геометрии (ширина ленты на полу).

### Параметры конфига

| Параметр | Default | Назначение |
|---|---|---|
| `vision_lat_yaw_gain` | `0.0001` | Yaw-коррекция из бокового смещения (cross-track → heading) |

### Что НЕ изменялось и почему
- **`imu_yaw_bias_window_sec`** — уже реализован корректно в `imu_io.py`.
- **Интегральный член для бокового смещения** — не добавлялся; yaw-коррекция решает ту же задачу без windup.
- **`imu_use_translation`** — оставлен `false`, т.к. IMU-трансляция без ZUPT сильно дрейфует.
- **Скорость** — оставлена константной до тех пор, пока не будет сенсорного источника.
