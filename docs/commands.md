# Команды проекта `awesome-feature-navigation`

## Назначение документа

Этот документ перечисляет реальные команды, которые можно запускать в этом репозитории, и объясняет:

- зачем нужна каждая команда;
- из какой директории ее запускать;
- какие флаги у нее есть;
- что именно произойдет после запуска;
- какие файлы появятся на выходе;
- какие типичные ошибки и ожидания есть у каждой команды.

Документ составлен по фактическому содержимому репозитория:

- `README.md`
- `awesome-feature-navigation-solution/README.md`
- `awesome-feature-navigation-solution/pyproject.toml`
- `awesome-feature-navigation-solution/awesome_feature_navigation/cli.py`
- `awesome-feature-navigation-solution/awesome_feature_navigation/tuner.py`
- `check-info.py`
- `fix_imu.py`

## Что в проекте есть на самом деле

В проекте есть следующие реальные точки входа:

| Команда | Тип | Назначение |
| --- | --- | --- |
| `uv run afn-run` | основной CLI | Построение 2D-траектории по видео и IMU |
| `uv run afn-tune` | основной CLI | Интерактивная настройка HSV-порогов для выделения цветной линии |
| `python -m awesome_feature_navigation.cli` | эквивалент `uv run afn-run` | Альтернативный запуск основного CLI через модуль |
| `python -m awesome_feature_navigation.tuner` | эквивалент `uv run afn-tune` | Альтернативный запуск тюнера через модуль |
| `python check-info.py` | вспомогательный скрипт | Быстрая проверка длительности видео и IMU |
| `python fix_imu.py` | вспомогательный скрипт | Нормализация timestamps IMU и сохранение исправленного CSV |

В проекте не объявлены:

- `Makefile`;
- `npm`-скрипты;
- `docker compose`;
- отдельная CLI-команда для тестов;
- отдельная CLI-команда для обучения модели.

Если нужна строгая формулировка: официальных CLI-команд у пакета ровно две, это `uv run afn-run` и `uv run afn-tune`.

## Директории и два способа запуска

Есть два нормальных сценария работы с репозиторием.

### Вариант 1. Запуск из каталога пакета

Это вариант, который использован в `README`.

```bash
cd awesome-feature-navigation-solution
uv sync
```

После этого можно запускать:

```bash
uv run afn-run ...
uv run afn-tune ...
```

### Вариант 2. Запуск из корня репозитория

Если пакет уже установлен через `uv sync` в каталоге `awesome-feature-navigation-solution`, можно запускать напрямую через окружение:

```bash
cd awesome-feature-navigation-solution
uv run afn-run ...
uv run afn-tune ...
```

или через `python -m`:

```bash
uv run python -m awesome_feature_navigation.cli ...
uv run python -m awesome_feature_navigation.tuner ...
```

### Что выбрать

Проще всего использовать вариант из `README`: перейти в `awesome-feature-navigation-solution` и всё запускать через `uv run`.

## Рекомендуемая последовательность команд

Если нужно пройти типичный рабочий цикл, команды обычно идут в таком порядке:

1. Установить зависимости через `uv sync`.
2. Проверить исходные данные через `python check-info.py`.
4. При необходимости подготовить IMU через `python fix_imu.py`.
5. Подобрать цвет линии через `uv run afn-tune`.
6. Запустить основную оценку траектории через `uv run afn-run`.
7. Просмотреть `csv`, `html` и при необходимости debug-видео.

Дальше подробно разобрана каждая команда.

## 1. Установка зависимостей

### Команда `uv sync`

```bash
cd awesome-feature-navigation-solution
uv sync
```

### Зачем нужна

Создаёт виртуальное окружение `.venv` и устанавливает все зависимости из `pyproject.toml`. Если есть `uv.lock`, использует зафиксированные версии.

### Флаги

Базовый вид без флагов подходит для большинства случаев.

| Флаг | Что делает |
| --- | --- |
| `--frozen` | Запрещает обновлять `uv.lock`, упавший с ошибкой если версии не совпадают |
| `--no-dev` | Не устанавливает dev-зависимости |

### Что ожидать

- Появится директория `.venv` с Python и установленными пакетами.
- Установятся зависимости: `numpy`, `opencv-python`, `matplotlib`, `plotly`, `pyyaml`, `pandas`.
- Код пакета ставится в editable-режиме (ссылкой на исходники).

### Типичные проблемы

- Если `uv` не установлен: `curl -LsSf https://astral.sh/uv/install.sh | sh`
- Повторный `uv sync` после изменения `pyproject.toml` автоматически подхватывает новые зависимости.

### Важное замечание

После `uv sync` **не нужно** активировать окружение вручную. Все команды запускаются через `uv run`, который автоматически использует `.venv`:

```bash
uv run afn-run ...
uv run afn-tune ...
```

## 3. Основная команда `uv run afn-run`

### Базовый вид команды

```bash
uv run afn-run \
  --video ../resources/Left_cam.mp4 \
  --imu ../resources/imu_fixed.csv \
  --imu-time-scale 1e-9 \
  --color red \
  --auto-color \
  --config examples/config.yaml \
  --out trajectory
```

### Что делает команда

Команда:

- читает видео;
- при наличии IMU читает и калибрует IMU CSV;
- выделяет линию по цвету в кадрах;
- оценивает направление линии и поперечное смещение;
- объединяет визуальные измерения и IMU;
- строит 2D-траекторию;
- сохраняет результат в `csv` и `html`.

### Когда ее запускать

Запускай `uv run afn-run`, когда:

- HSV-пороги уже подходят для твоего видео;
- у тебя есть видео и, возможно, IMU;
- ты хочешь получить итоговую траекторию в файлах.

### Синтаксис

```bash
uv run afn-run --video VIDEO [дополнительные флаги]
```

### Полный разбор всех CLI-флагов `uv run afn-run`

#### Флаг `-h`, `--help`

```bash
uv run afn-run --help
```

Назначение:

- печатает справку;
- не запускает обработку видео.

Ожидание:

- никаких выходных файлов;
- только текст справки в терминале.

#### Флаг `--video VIDEO`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4
```

Назначение:

- указывает путь к видео;
- это обязательный флаг;
- ожидается `.mp4`, но фактически OpenCV пытается открыть любой поддерживаемый видеоконтейнер.

Что передавать:

- относительный путь;
- абсолютный путь;
- путь к реально существующему файлу.

Что будет, если не передать:

- `argparse` завершит программу с ошибкой, потому что флаг обязательный.

Что будет, если путь неправильный:

- программа завершится с ошибкой `Cannot open video: ...`.

Что ожидать при корректном запуске:

- начнется покадровая обработка видео;
- прогресс-бар не выводится;
- в конце будут сохранены результаты.

#### Флаг `--imu IMU`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --imu ../resources/imu_fixed.csv
```

Назначение:

- задает путь к CSV с IMU.

Что меняет:

- если флаг передан, команда подключает IMU к расчету yaw;
- если флаг не передан, проект работает в demo-режиме без IMU.

Что происходит без `--imu`:

- траектория все равно строится;
- движение вперед берется из `forward_speed_mps` в YAML;
- yaw корректируется только по видео, без IMU-прединтеграции.

Что ожидать:

- CSV должен содержать колонку времени и оси акселерометра и гироскопа;
- имена колонок могут быть гибкими, но смысл колонок должен быть корректным.

#### Флаг `--imu-time-scale IMU_TIME_SCALE`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --imu ../resources/imu_fixed.csv --imu-time-scale 1e-9
```

Назначение:

- умножает timestamps IMU на заданный коэффициент.

Зачем нужен:

- многие IMU-файлы хранят время не в секундах, а в наносекундах;
- коду нужны секунды;
- `1e-9` переводит наносекунды в секунды.

Что ожидать:

- временная шкала IMU станет совместима со временем видео.

Типичный правильный сценарий:

- если timestamps вида `1772267134029791765`, нужен именно `1e-9`.

Что будет, если масштаб неверный:

- IMU и видео окажутся в разных временных шкалах;
- yaw и траектория будут бессмысленными или сильно искажёнными.

#### Флаг `--imu-gyro-scale IMU_GYRO_SCALE`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --imu ../resources/imu_fixed.csv --imu-gyro-scale 0.017453292519943295
```

Назначение:

- умножает значения гироскопа на заданный коэффициент до начала интеграции.

Зачем нужен:

- если гироскоп в `deg/s`, а модели нужен `rad/s`, надо перевести единицы измерения.

Типичный пример:

- `0.017453292519943295` это `pi / 180`;
- такой коэффициент переводит градусы/сек в радианы/сек.

Что ожидать:

- при правильном коэффициенте yaw перестанет быть заведомо неправильным по масштабу.

Что будет при ошибочном коэффициенте:

- повороты будут слишком сильными или слишком слабыми.

#### Флаг `--config CONFIG`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --config examples/config.yaml
```

Назначение:

- загружает YAML-конфиг с параметрами детекции, сглаживания, fusion и дополнительных режимов.

Что ожидать:

- если файл существует и корректен, значения будут подхвачены;
- если файла нет, будет ошибка чтения файла;
- если YAML пустой, будет использован пустой словарь и дефолты из кода.

Что важно понимать:

- часть поведения задается только через YAML, а не через CLI;
- CLI-флаги могут перекрывать поля конфига.

#### Флаг `--color {red,blue,green,yellow,white}`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --color red
```

Назначение:

- выбирает целевой пресет цвета линии.

Поддерживаемые значения:

- `red`
- `blue`
- `green`
- `yellow`
- `white`

Что делает под капотом:

- если в YAML не задан или не подходит `hsv_ranges`, будет использован пресет для указанного цвета;
- если `hsv_ranges` явно заданы в YAML, они имеют приоритет, но `target_color` все равно влияет на автонастройку и логику тюнера.

Что ожидать:

- нужно выбирать цвет, соответствующий реальной линии в видео;
- неверный цвет почти всегда ломает детекцию.

#### Флаг `--auto-color`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --color red --auto-color
```

Назначение:

- включает авто-подстройку HSV-порогов под освещение конкретного видео.

Что это значит:

- код берет базовые диапазоны HSV;
- затем слегка поджимает или расширяет их по статистике пикселей в рабочей области кадра.

Когда помогает:

- если цвет линии тот же, но освещение отличается;
- если пресет близок к правильному, но не попадает в конкретную запись.

Когда не помогает:

- если выбран вообще не тот цвет;
- если линия почти не видна;
- если сцена слишком шумная.

Что ожидать:

- детекция может стать заметно стабильнее;
- иногда, наоборот, автонастройка может испортить уже хорошо подобранные ручные пороги.

#### Флаг `--imu-use-translation`

Пример:

```bash
uv run afn-run \
  --video ../resources/Left_cam.mp4 \
  --imu ../resources/imu_fixed.csv \
  --imu-time-scale 1e-9 \
  --imu-use-translation
```

Назначение:

- включает использование прединтегрированного смещения IMU по XY вместо постоянной скорости `forward_speed_mps`.

Что меняет:

- без этого флага движение вперед идет как `forward_speed_mps * dt`;
- с этим флагом XY-сдвиг берется из IMU `delta_p`.

Когда использовать:

- только если IMU хорошо откалиброван по осям, bias, гравитации и единицам измерения.

Риск:

- на шумных данных акселерометра этот режим может дать большой дрейф и сделать траекторию хуже.

Практическое ожидание:

- это режим для эксперимента и диагностики, а не всегда лучший дефолт.

#### Флаг `--out OUT`

Пример:

```bash
uv run afn-run --video ../resources/Left_cam.mp4 --out trajectory
```

Назначение:

- задает префикс выходных файлов.

Примеры:

- `--out trajectory` создаст `trajectory.csv` и `trajectory.html`;
- `--out results/run1` создаст `results/run1.csv` и `results/run1.html`, если директория существует;
- `--out /tmp/test` создаст файлы в `/tmp`.

Что ожидать:

- расширения `.csv` и `.html` добавляются автоматически;
- если включены дополнительные режимы, будут созданы и дополнительные файлы с тем же префиксом.

#### Флаг `--save-debug`

Пример:

```bash
uv run afn-run \
  --video ../resources/Left_cam.mp4 \
  --imu ../resources/imu_fixed.csv \
  --imu-time-scale 1e-9 \
  --config examples/config.yaml \
  --out trajectory \
  --save-debug
```

Назначение:

- сохраняет debug-видео поверх исходного видео.

Что будет в debug-видео:

- цветная маска выделенной линии;
- отметка `bottom_x`;
- текст с `t`, `x`, `y`, `yaw`.

Имя выходного файла:

- `PREFIX_debug.mp4`

Что ожидать:

- выполнение может занять больше времени;
- размер результата может быть большим, особенно для длинного ролика.

### Какие файлы создает `uv run afn-run`

Минимальный набор:

- `PREFIX.csv`
- `PREFIX.html`

Если задан `--save-debug`:

- `PREFIX_debug.mp4`

Если в YAML включен `loop_average: true`:

- `PREFIX_raw.csv`
- `PREFIX_raw.html`
- `PREFIX_laps.csv`
- `PREFIX_laps.html`

### Что лежит в `PREFIX.csv`

Это таблица с колонками:

- `t`
- `x`
- `y`
- `yaw`

То есть на выходе ты получаешь временной ряд траектории.

### Что лежит в `PREFIX.html`

Это интерактивный Plotly-график траектории:

- можно приближать;
- можно панорамировать;
- можно наводить курсор и смотреть координаты и время.

### Пример минимального рабочего запуска без IMU

```bash
uv run afn-run \
  --video ../resources/Left_cam.mp4 \
  --color red \
  --auto-color \
  --config examples/config.yaml \
  --out demo_run
```

Когда это полезно:

- если IMU еще не подготовлен;
- если сначала хочешь проверить только визуальную часть.

Что ожидать:

- траектория будет построена без IMU;
- линейная скорость будет взята из `forward_speed_mps` конфига.

### Пример стандартного запуска с IMU

```bash
uv run afn-run \
  --video ../resources/Left_cam.mp4 \
  --imu ../resources/imu_fixed.csv \
  --imu-time-scale 1e-9 \
  --color red \
  --auto-color \
  --config examples/config.yaml \
  --out trajectory
```

Это основной сценарий для `Left_cam.mp4`.

### Пример запуска для `Right_cam`

```bash
uv run afn-run \
  --video ../resources/Right_cam.mp4 \
  --imu ../resources/imu_fixed.csv \
  --config examples/right_cam.yaml \
  --out right_cam
```

Что ожидать:

- будут использованы параметры IMU и loop-усреднения из `examples/right_cam.yaml`;
- если в конфиге включен `loop_average`, появятся дополнительные `_raw` и `_laps` файлы.

### Типичные ошибки при `uv run afn-run`

#### Ошибка: видео не открывается

Причина:

- неверный путь;
- файл поврежден;
- OpenCV не может открыть контейнер.

Что делать:

- проверить путь;
- убедиться, что файл существует;
- попробовать открыть видео отдельно.

#### Ошибка: IMU не читается

Причина:

- неверный путь к CSV;
- нет обязательных колонок;
- CSV поврежден.

Что делать:

- проверить файл через `python check-info.py`;
- убедиться, что есть время, `ax/ay/az`, `gx/gy/gz`.

#### Результат есть, но траектория явно неверная

Частые причины:

- не тот `--color`;
- неверный `--imu-time-scale`;
- неверный `--imu-gyro-scale`;
- плохие HSV-пороги;
- слишком агрессивные gain-ы в YAML;
- использование `--imu-use-translation` на некалиброванном IMU.

## 4. Интерактивная команда `uv run afn-tune`

### Базовый вид команды

```bash
uv run afn-tune \
  --video ../resources/Left_cam.mp4 \
  --config examples/config.yaml \
  --color red \
  --auto-color
```

### Что делает команда

Команда открывает интерактивное окно OpenCV и позволяет подбирать параметры детекции линии на реальном видео.

В окне ты видишь:

- исходный кадр;
- бинарную маску линии;
- centerline;
- текущий цвет;
- состояние `auto`;
- текущие значения `angle` и `bottom_x`.

### Когда ее запускать

Запускай `uv run afn-tune`, когда:

- линия плохо выделяется;
- ты не уверен, какой цветовой пресет нужен;
- нужно вручную настроить HSV;
- нужно сохранить рабочий YAML для конкретной камеры или видео.

### Полный разбор всех CLI-флагов `uv run afn-tune`

#### Флаг `-h`, `--help`

```bash
uv run afn-tune --help
```

Назначение:

- печатает справку и завершает выполнение.

#### Флаг `--video VIDEO`

Пример:

```bash
uv run afn-tune --video ../resources/Left_cam.mp4
```

Назначение:

- обязательный путь к видео, которое будет проигрываться в тюнере.

Что ожидать:

- видео откроется в цикле;
- если дойдёт до конца, проигрывание начнется заново.

Если путь неверный:

- программа завершится с ошибкой `Cannot open video: ...`.

#### Флаг `--config CONFIG`

Пример:

```bash
uv run afn-tune --video ../resources/Left_cam.mp4 --config examples/config.yaml
```

Назначение:

- путь к YAML, из которого читаются стартовые параметры и в который потом сохраняются настройки.

Что ожидать:

- если файл существует, текущие значения будут загружены;
- если файла нет, старт будет с дефолтами;
- по нажатию `s` настройки будут записаны в этот путь.

Важно:

- тюнер реально перезаписывает YAML-файл;
- если не хочешь портить существующий конфиг, передай новый путь, например `examples/my_left_cam.yaml`.

#### Флаг `--color {red,blue,green,yellow,white}`

Пример:

```bash
uv run afn-tune --video ../resources/Left_cam.mp4 --color red
```

Назначение:

- задает стартовый цветовой пресет.

Что ожидать:

- в трекбары загрузятся пороги для выбранного цвета;
- при этом их можно будет потом менять вручную.

#### Флаг `--auto-color`

Пример:

```bash
uv run afn-tune --video ../resources/Left_cam.mp4 --config examples/config.yaml --color red --auto-color
```

Назначение:

- включает режим автоподстройки HSV в самом тюнере.

Что ожидать:

- `Auto` будет включен сразу после старта;
- состояние можно потом изменить трекбаром.

### Управление в окне `uv run afn-tune`

Клавиши:

- `q` закрыть окно и завершить программу;
- `s` сохранить текущие настройки в YAML;
- `r` сбросить текущие пороги к пресету выбранного цвета;
- `1` выбрать `red`;
- `2` выбрать `blue`;
- `3` выбрать `green`;
- `4` выбрать `yellow`;
- `5` выбрать `white`.

### Что именно сохраняет тюнер

При нажатии `s` в YAML сохраняются:

- `target_color`
- `auto_color_tune`
- `hsv_ranges`

То есть тюнер не сохраняет всю физику траектории, а только настройки цветовой детекции.

### Что ожидать после запуска

- Откроется графическое окно OpenCV.
- В headless-среде, без GUI или без доступа к окнам, тюнер работать не будет.
- Никакие файлы не сохраняются автоматически.
- YAML обновляется только после нажатия `s`.

### Практический сценарий использования

```bash
uv run afn-tune \
  --video ../resources/Right_cam.mp4 \
  --config examples/right_cam.yaml \
  --color blue
```

Этот сценарий особенно полезен, когда надо сделать отдельный конфиг под конкретную камеру.

## 5. Вспомогательная команда `python check-info.py`

### Команда

```bash
python check-info.py
```

### Где запускать

Из корня репозитория:

```bash
cd awesome-feature-navigation
python check-info.py
```

### Что делает

Скрипт:

- проверяет, существует ли `./resources/Left_cam.mp4`;
- открывает видео через OpenCV;
- печатает FPS, число кадров и длительность;
- читает `./resources/imu_data.csv`;
- печатает число строк IMU;
- находит колонку времени;
- оценивает длительность IMU-записи;
- если timestamps слишком большие, рекомендует `--imu-time-scale 1e-9`.

### Важное ограничение

У этой команды нет CLI-флагов.

Скрипт использует жестко зашитые пути:

- `./resources/Left_cam.mp4`
- `./resources/imu_data.csv`

Если ты хочешь проверить другие файлы, сейчас нужно редактировать сам скрипт.

### Что ожидать

- Никакие файлы не создаются.
- Это только диагностический вывод в терминал.

### Когда запускать

Запускай перед `uv run afn-run`, если хочешь понять:

- насколько совпадают длительности видео и IMU;
- в каких единицах времени может быть IMU;
- есть ли вообще смысл сразу пробовать `--imu-time-scale 1e-9`.

### Пример типичного вывода

Скрипт может показать примерно такую информацию:

- FPS видео;
- число кадров;
- длительность видео;
- число IMU-измерений;
- первый и последний timestamp;
- предупреждение, что timestamps выглядят как наносекунды.

### Что означают результаты

Если скрипт пишет, что timestamps очень большие, это почти всегда означает, что:

- IMU хранит время в наносекундах;
- `uv run afn-run` нужно запускать с `--imu-time-scale 1e-9`.

## 6. Вспомогательная команда `python fix_imu.py`

### Команда

```bash
python fix_imu.py
```

### Где запускать

Из корня репозитория.

### Что делает

Скрипт:

- читает `./resources/imu_data.csv`;
- пытается найти колонку времени;
- переименовывает ее в `timestamp_ns`;
- сортирует строки по времени;
- вычитает стартовый timestamp;
- сохраняет результат в `./resources/imu_fixed.csv`.

### Важное ограничение

У этой команды тоже нет CLI-флагов.

Пути внутри скрипта жестко зашиты:

- вход: `./resources/imu_data.csv`
- выход: `./resources/imu_fixed.csv`

### Что ожидать

- будет создан или перезаписан файл `resources/imu_fixed.csv`;
- timestamps в выходном CSV будут начинаться с нуля;
- выходной файл удобно подавать в `uv run afn-run`.

### Когда запускать

Запускай `python fix_imu.py`, если:

- исходный IMU-файл использует абсолютные timestamps;
- нужно привести шкалу времени к старту от нуля;
- ты хочешь получить промежуточный CSV, с которым проще работать дальше.

### Что не делает этот скрипт

Скрипт не:

- переводит наносекунды в секунды;
- не меняет единицы гироскопа;
- не калибрует bias;
- не проверяет качество данных.

Он делает только одно: нормализует время так, чтобы оно начиналось с нуля.

### Что делать после него

После `python fix_imu.py` следующий типичный шаг:

```bash
uv run afn-run \
  --video ../resources/Left_cam.mp4 \
  --imu ../resources/imu_fixed.csv \
  --imu-time-scale 1e-9 \
  --config examples/config.yaml \
  --color red \
  --auto-color \
  --out trajectory
```

## 8. Параметры в YAML, которые влияют на команды

Пользователь спрашивал именно про команды и флаги, но для этого проекта этого недостаточно: значительная часть поведения управляется полями YAML.

Ниже перечислены ключевые поля, потому что без них невозможно строго понять, чего ожидать от `uv run afn-run`.

## 8.1. Поля детекции линии

### `target_color`

Назначение:

- цветовой пресет по умолчанию.

Допустимые значения:

- `red`
- `blue`
- `green`
- `yellow`
- `white`

### `auto_color_tune`

Назначение:

- включает автоподстройку HSV из конфига, даже если флаг `--auto-color` не был передан.

### `hsv_ranges`

Пример:

```yaml
hsv_ranges:
  - [0, 92, 51, 10, 235, 255]
  - [158, 120, 70, 151, 247, 255]
```

Назначение:

- ручные HSV-диапазоны.

Формат каждого диапазона:

- `H_low, S_low, V_low, H_high, S_high, V_high`

Если диапазонов два:

- это полезно для красного цвета, у которого часто два диапазона hue.

### `centerline_smooth_window`

Назначение:

- сглаживает centerline внутри одного кадра.

Эффект:

- больше значение дает более гладкую линию;
- слишком большое значение может "замыть" геометрию.

### `resize_width`

Назначение:

- задает ширину, к которой приводятся кадры перед детекцией.

Если не задано:

- код использует дефолт `640`.

## 8.2. Поля визуально-инерциального fusion

### `vision_yaw_gain`

Назначение:

- коэффициент визуальной поправки к yaw по углу линии.

### `vision_yaw_max_correction`

Назначение:

- ограничивает максимальную визуальную yaw-поправку.

### `vision_lateral_gain`

Назначение:

- коэффициент поперечной коррекции по `bottom_x`.

### `vision_smoothing_frames`

Назначение:

- число кадров для EMA-сглаживания `angle_rad` и `bottom_x`.

### `forward_speed_mps`

Назначение:

- постоянная скорость вперед, используемая, если не включена IMU-трансляция.

### `max_frames`

Назначение:

- ограничивает число обрабатываемых кадров.

Что значит значение `0`:

- обрабатывать все кадры до конца видео.

## 8.3. Поля IMU-калибровки

### `imu_time_scale`

Назначение:

- YAML-аналог `--imu-time-scale`.

### `imu_gyro_scale`

Назначение:

- YAML-аналог `--imu-gyro-scale`.

### `imu_accel_axes`

Назначение:

- перестановка и инверсия осей акселерометра.

Пример формата:

```yaml
imu_accel_axes: [x, y, z]
```

или:

```yaml
imu_accel_axes: [x, -z, y]
```

### `imu_gyro_axes`

Назначение:

- перестановка и инверсия осей гироскопа.

### `imu_gyro_bias`

Назначение:

- постоянное смещение гироскопа, которое надо вычесть.

### `imu_align_gravity`

Назначение:

- поворачивает данные так, чтобы средняя гравитация смотрела в `[0, 0, -1]`.

### `imu_yaw_axis`

Назначение:

- указывает, какая ось после калибровки отвечает за yaw.

Примеры:

- `z`
- `-z`
- `x`

### `imu_yaw_bias`

Назначение:

- ручной bias для выбранной yaw-оси.

### `imu_yaw_bias_window_sec`

Назначение:

- если задано больше нуля, bias оценивается автоматически по началу записи на этом интервале.

### `imu_yaw_gain`

Назначение:

- масштаб yaw-скорости после выделения нужной оси.

### `imu_yaw_only`

Назначение:

- если `true`, остальные оси угловой скорости зануляются, и используется только yaw-компонента.

### `imu_use_translation`

Назначение:

- YAML-аналог `--imu-use-translation`.

## 8.4. Поля loop-усреднения траектории

Эти параметры особенно важны для сценариев, где запись содержит несколько проходов по одной и той же петле.

### `loop_average`

Назначение:

- включает канонизацию периодической траектории.

Если `false`:

- итогом будет сырая траектория.

Если `true`:

- итогом может стать усредненная замкнутая петля и дополнительные diagnostic-файлы.

### `loop_period_sec`

Назначение:

- ожидаемый период одного круга или одной петли.

### `loop_samples`

Назначение:

- число сэмплов на одну петлю.

### `loop_min_fraction`

Назначение:

- минимальная доля периода, чтобы фрагмент считался валидным кругом.

### `loop_fourier_harmonics`

Назначение:

- степень Fourier-сглаживания для канонической петли.

### `loop_refine_iterations`

Назначение:

- число итераций уточнения шаблона петли.

### `loop_outlier_sigma`

Назначение:

- порог выбросов по robust-z для отбора плохих кругов.

### `loop_similarity_align`

Назначение:

- разрешает не только поворот и перенос, но и масштабирование при совмещении кругов.

### `loop_phase_search_fraction`

Назначение:

- насколько далеко искать оптимальный фазовый сдвиг внутри круга.

### `loop_anchor_search_fraction`

Назначение:

- радиус поиска временных границ кругов.

### `loop_projection_search_fraction`

Назначение:

- радиус поиска точки проекции на канонический контур.

### `loop_average_statistic`

Допустимые значения:

- `mean`
- `median`

Назначение:

- определяет, как строится представитель круга из нескольких проходов.

### `loop_spline_control_points`

Назначение:

- число контрольных точек сплайна канонической петли.

### `loop_start_anchor`

Примеры:

- `bottom_left`
- `top_left`
- `top_right`
- `bottom_right`
- `left`
- `right`
- `top`
- `bottom`

Назначение:

- задает, где на канонической петле считать начало траектории.

## 10. Что важно помнить перед запуском

### Про `uv run afn-run`

- Это не интерактивная команда.
- Она не показывает progress bar.
- Она просто работает и в конце печатает `Saved: ...`.

### Про `uv run afn-tune`

- Это интерактивная GUI-команда.
- Она не подходит для чисто headless-сервера без окон.
- YAML сохраняется только вручную по `s`.

### Про `check-info.py`

- Это диагностический скрипт.
- Он не меняет данные.
- Он не принимает аргументы.

### Про `fix_imu.py`

- Это подготовительный скрипт.
- Он создает или перезаписывает `resources/imu_fixed.csv`.
- Он не делает полноценную IMU-калибровку, только нормализует время.

### Про конфиги

- В этом проекте реальные результаты сильно зависят от YAML.
- Даже при одинаковой CLI-команде два разных YAML могут давать очень разную траекторию.

| Команда | Когда запускать | Что получишь |

| `python fix_imu.py` | Если timestamps IMU абсолютные | `resources/imu_fixed.csv` |

## 12. Строгий итог

Если отвечать максимально строго и коротко:

- у пакета есть две основные команды: `uv run afn-run` и `uv run afn-tune`;
- у репозитория есть два вспомогательных скрипта: `python check-info.py` и `python fix_imu.py`;
- всё остальное — это либо `uv sync` для подготовки окружения, либо эквиваленты через `uv run python -m`.

Если нужно начать прямо сейчас с рабочего минимального сценария, используй такой порядок:

```bash
cd awesome-feature-navigation
python check-info.py
python fix_imu.py
cd awesome-feature-navigation-solution
uv sync
uv run afn-tune --video ../resources/Left_cam.mp4 --config examples/config.yaml --color red --auto-color
uv run afn-run --video ../resources/Left_cam.mp4 --imu ../resources/imu_fixed.csv --imu-time-scale 1e-9 --color red --auto-color --config examples/config.yaml --out trajectory
```

Это самый прямой и понятный маршрут по текущему проекту.
